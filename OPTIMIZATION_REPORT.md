**샘플 영상:**

## 단계별 세부 분석

### 1. ffmpeg split 필터 (Opt 1) — 가장 큰 효과

동일 영상을 **2회 디코딩 → 1회 디코딩**으로 전환.

```
변경 전: ffmpeg (속력 크롭) → ffmpeg (핸들 크롭)  — 별도 2회 실행
변경 후: ffmpeg -filter_complex split  — 1회 실행으로 두 스트림 동시 출력
```

| 변경 후 (split) | — | — | **152.5 초** |
| **절감** | | | **153.3 초 (▼ 50.1%)** |

### 4. 영상 단위 병렬 처리 (Opt 4, `--workers`)

`ProcessPoolExecutor`를 이용해 여러 영상을 동시에 처리. 단일 영상에는 효과 없음.

---

#### 4-A. 초기 실험 (3개 영상 — 기록.mp4 포함) — **무효**

기록.mp4가 유효하지 않은 영상임이 확인되어 삭제됨. 해당 실험 결과는 참고 기록으로만 남김.

```
[무효] 실측 — 기록.mp4 + 2.mp4 + 3.mp4 (전체 파이프라인 포함)

| 처리 방식 | 소요시간 | 단축률 |
| workers=1 | 469.9 초 |  —     |
| workers=2 | 390.9 초 | ▼16.8% |
```

workers=2 실행 타임라인:
  0:00  기록.mp4(53초) + 2.mp4 병렬 시작 → 1:38에 기록.mp4 완료로 슬롯 조기 해제
  1:38  3.mp4 시작 → 2.mp4와 4분 01초 overlap
  5:39  2.mp4 완료 → 3.mp4 단독 처리 51초
  6:30  종료

기록.mp4(53초) ≪ 2.mp4(200초) 처리 시간 편차로 workers=2 슬롯 하나가 조기 해제됨 → 병렬 효율 저하.

---

#### 4-B. 재실험 (2개 영상 — 2.mp4 + 3.mp4) — **유효**

기록.mp4 삭제 후 **2.mp4 + 3.mp4** 2개 영상으로 재실험.

**측정 범위:** classify_sevenseg + classify_steering + export_excel 단계 (ffmpeg 프레임은 이미 추출된 상태, OS 디스크 캐시 warm)  
**환경 변경:** 실험 과정에서 pandas 1.5.0 → 2.3.3 업그레이드 (numpy 2.0.2 호환성 문제 해소)

```
python3.9 run_all.py --source source/sample_name --workers 1   # 순차
python3.9 run_all.py --source source/sample_name --workers 2   # 병렬
```

| 처리 방식 | 실측 소요시간 | 단축률 | CPU 활용 |
|---|---:|---:|---:|
| 순차 (`workers=1`) | **57.5 초** | — | 106% |
| 병렬 (`workers=2`) | **30.8 초** | **▼ 46.4%** | 205% |

**workers=2 실행 타임라인:**

```
0:00   2.mp4 (9,010 프레임) + 3.mp4 (10,871 프레임) 동시 시작
0:26   2.mp4 완료
0:29   3.mp4 완료  →  총 30.8초
```

> **이전 실험(▼16.8%)과의 차이:** 기록.mp4(~53초)를 제거하고 처리 시간이 유사한 영상 2개만 남기자 workers=2 슬롯이 시작~종료까지 풀 가동됨. 비슷한 크기의 영상이 workers 슬롯을 꽉 채울수록 이상적인 병렬 효율(이론 최대 ≈ ▼50%)에 수렴한다.  
> 영상 수 ≥ workers×2 이상이거나 영상 길이가 비슷한 경우 큰 단축 효과를 기대할 수 있음.  
> 권장: 게이밍 PC `--workers 4~6`, 일반 PC `--workers 2~3`.

## 완료된 추가 최적화

### Opt 6: overlay=False 시 순차 로딩 복귀 (classify_sevenseg / classify_steering)

`_load_frames_parallel()` 호출을 `overlay` 플래그로 분기 처리.

```python
# 변경 전 (항상 ThreadPoolExecutor)
imgs = _load_frames_parallel(paths)

# 변경 후
if overlay:
    imgs = _load_frames_parallel(paths)   # 병렬: I/O 절감 > 풀 초기화 오버헤드
else:
    imgs = [_load_one_frame(p) for p in paths]  # 순차: 풀 오버헤드 없음
```

**배경:** 58×19 px 소형 이미지에서 `ThreadPoolExecutor` 초기화 비용(~300 ms)이 I/O 병렬화 절감을 초과함. overlay=False(일반 사용 케이스)에서 오히려 느려지는 역효과 발생.

| 파일 | overlay=False 효과 |
|---|---|
| classify_sevenseg.py | 풀 오버헤드 제거 → 순수 CPU 처리만 |
| classify_steering.py | 풀 오버헤드 제거 → 순수 CPU 처리만 |
